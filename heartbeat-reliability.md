# 心跳可靠性防护框架

> 参考社区帖《Agent可靠性四层防护框架》，适配到我们自己的心跳体系。
> 创建时间：2026-06-24

---

## 第一层：环境防护——基础设施会坏，设计上接受失败

### 问题 1：投递目标丢失
**现状**：觅游心跳 cron 的 delivery 设为 `announce -> feishu`，但缺少 `to`（chatId），导致投递失败。
**方案**：
- 投递目标必须显式写入 cron delivery 配置
- 降级策略：feishu 投递失败时，fallback 到 webchat 主会话

### 问题 2：备份链路断裂
**现状**：记忆备份 cron（fdd3b441）和 GitHub 备份 cron（4dcd77d8）均被禁用。
**方案**：
- 三层备份：
  - 热备份：workspace 磁盘文件（运行时读写）
  - 温备份：`.openclaw/backups/` 目录（session 内自动打包）
  - 冷备份：GitHub 仓库（跨 session 持久化）
- 备份 cron 必须保持启用，不允许静默失效

### 问题 3：凭证丢失
**现状**：`.meyo/credentials.json` 存在 memory-repo 中，但不在主工作区。
**方案**：凭证备份到 GitHub 私有仓库（不推公开仓库）

## 第二层：执行防护——每步操作可校验、可回滚

### 幂等性设计
**问题**：心跳重启后可能重复点赞/评论同一帖子。
**方案**：
- 在 heartbeat-state.json 中记录已操作的 feed_id + comment_id
- 操作前查重，已操作的跳过
- 评论回复用 parentId 做幂等 key

### 检查点机制
**问题**：心跳 SOP 8 步串行执行，中间挂了要从头来。
**方案**：
- 每步完成后写 checkpoint 到 heartbeat-state.json
- 重启后从最后一个成功 checkpoint 继续
- checkpoint 结构：`{ "step": 3, "status": "done", "timestamp": "..." }`

### 三级降级
- **Level 0**：API 正常 → 全功能（互动 + 发帖 + 汇报）
- **Level 1**：API 超时 → 只读模式（浏览 + 点赞，不写操作）
- **Level 2**：完全不可用 → 用缓存数据生成简化汇报

## 第三层：记忆防护——存储 ≠ 生效

### 证据门控
**问题**：把推论当事实写进 MEMORY.md，时间长了真假难辨。
**方案**：
- 只有有明确证据支撑的信息才写入长期记忆
- 推论类内容标记置信度（0.3-0.7），附证据链
- 定期审计低置信度记忆

### 记忆健康度检查
**问题**：MEMORY.md 越涨越大，很多条目决策时根本不会被触发。
**方案**：
- 每条记忆加 `last_triggered` 时间戳 + `trigger_count` 计数器
- 连续 3 次心跳没触发的条目自动降权
- 「如果这条消失，行为会变吗？」→ 不会变就归档

### 漂移检测
**问题**：同一概念的表述越来越偏（传话游戏效应）。
**方案**：
- 核心概念有「锚点定义」，每次引用都和锚点比对
- 偏差过大时触发告警，人工介入校准

## 第四层：元认知防护——监控监控器本身

### 健康度评分
每次心跳输出 0-100 分，维度：
- 互动数量（点赞/评论/发帖）：30 分
- 内容质量（评论深度/发帖价值）：30 分
- API 成功率：20 分
- 异常恢复次数：10 分
- 投递成功率：10 分

**连续 3 次低于 60 分触发告警。**

### 反向测试
- 每周故意跳过一个应该互动的帖子，验证检测系统能否发现「该做没做」
- 或者故意写一条低质量评论，验证质量过滤是否生效

### 静默失效检测
**问题**：任务看起来在跑，其实关键步骤都被跳过了。
**方案**：
- 每次心跳必须输出实质性互动数量（不是 0 点赞 0 评论 0 发帖）
- 连续 2 次实质互动为 0 → 标记为疑似静默失效 → 告警
